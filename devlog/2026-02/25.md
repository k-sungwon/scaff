### 1. 서버/클라이언트 컴포넌트 분리 아키텍처

**결정**: Next.js 15의 서버 컴포넌트를 최대한 활용하고, 상호작용이 필요한 부분만 클라이언트 컴포넌트로 분리

**이유**:

- 초기 페이지 로딩 속도 최적화
- SEO 향상 (문제 메타데이터가 서버에서 렌더링)
- Node.js API (`fs` 모듈) 사용 가능
- 번들 사이즈 감소 (서버 코드는 클라이언트로 전송되지 않음)

**트레이드오프**:

- 장점: 성능 향상, 초기 로딩 속도 빠름, SEO 최적화
- 단점: 컴포넌트 경계 설정에 주의 필요, 함수를 props로 전달 불가

**코드 예시**:

### 2. 페이지 이동 없는 대화형 UI

**결정**: 문제 선택 시 페이지 이동(`/problems/[id]`) 대신 같은 페이지에서 컴포넌트를 조건부 렌더링

**이유**:

- 사용자 경험 향상 (페이지 깜빡임 없음)
- Claude와 같은 LLM 도구의 대화형 인터페이스 모방
- 문제 목록 ↔ 해설 간 빠른 전환
- 상태 유지 용이 (어떤 문제를 봤는지 등)

**트레이드오프**:

- 장점: 부드러운 UX, 빠른 전환, 상태 유지
- 단점: URL이 변경되지 않아 공유 링크 생성 불가 (추후 query string으로 해결 가능)

**코드 예시**:

```typescript
// components/home/HomeClient.tsx
export function HomeClient({ initialProblems, problemDataMap }: HomeClientProps) {
  const [selectedProblem, setSelectedProblem] = useState<number | null>(null);
  const selectedProblemData = selectedProblem !== null ? problemDataMap[selectedProblem] : null;

  return (
    <>
      {/* 문제 선택 그리드 - 항상 표시 */}
      <div className={`...transition-all ${selectedProblem !== null ? 'mb-8' : ''}`}>
        {/* 문제 번호 버튼들 */}
        <button onClick={() => setSelectedProblem(problem.number)}>
          {problem.number}
        </button>
      </div>

      {/* 문제 뷰어 - 조건부 렌더링 */}
      {selectedProblem !== null && selectedProblemData && (
        <ProblemViewer
          meta={selectedProblemData.meta}
          clues={selectedProblemData.clues}
          steps={selectedProblemData.steps}
          onClose={() => setSelectedProblem(null)}
        />
      )}
    </>
  );
}
```

### 3. 메시지 기반 대화형 인터페이스

**결정**: 단서/풀이를 "User 요청 → Assistant 응답" 형태의 메시지 배열로 표현

**이유**:

- Claude와 같은 친숙한 대화형 UX
- 단계적 학습에 적합 (사용자가 원할 때만 다음 단서 공개)
- 시간 순서대로 내용이 쌓이는 자연스러운 플로우

**트레이드오프**:

- 장점: 직관적인 UX, 학습 속도 조절 가능, 친숙한 인터페이스
- 단점: 메시지 상태 관리 복잡도 증가, 메모리 사용량 증가 (메시지가 많아질수록)

### 4. 클라이언트 사이드 MDX 컴파일

**결정**: 서버가 아닌 클라이언트에서 `next-mdx-remote`의 `serialize()` 사용

**이유**:

- 서버 컴포넌트에서 모든 MDX를 컴파일하면 초기 로딩 시간 증가
- 사용자가 단서/풀이를 열 때만 컴파일 → 필요 시점에 컴파일
- 단계적 공개 UX와 잘 맞음

**트레이드오프**:

- 장점: 초기 로딩 속도 빠름, 필요할 때만 리소스 사용
- 단점: 클라이언트 번들 크기 증가 (remark/rehype 플러그인 포함), 컴파일 시 짧은 로딩 상태 필요

**코드 예시**:

```typescript
// components/problem/ConversationView.tsx
function SolutionStep({ stepNumber, title, content }: StepProps) {
  const [mdxSource, setMdxSource] = useState<MDXRemoteSerializeResult | null>(null);
  const [isCompiling, setIsCompiling] = useState(true);

  useEffect(() => {
    // 클라이언트에서 MDX 컴파일
    serialize(content, {
      mdxOptions: {
        remarkPlugins: [remarkMath],
        rehypePlugins: [rehypeKatex as any],
      },
    })
      .then((compiled) => {
        setMdxSource(compiled);
        setIsCompiling(false);
      })
      .catch((error) => {
        console.error("MDX compilation error:", error);
        setIsCompiling(false);
      });
  }, [content]);

  return (
    <div className="bg-white rounded-2xl shadow-lg p-6">
      <h3>풀이 {stepNumber}: {title}</h3>
      <div className="prose prose-sm max-w-none">
        {isCompiling ? (
          <p className="text-gray-500">로딩 중...</p>
        ) : mdxSource ? (
          <MDXRemote {...mdxSource} />
        ) : null}
      </div>
    </div>
  );
}
```

## 발생한 이슈

### 이슈 1: Tailwind CSS 4 커스텀 컬러 오류

**증상**:

```
Error evaluating Node.js code
Cannot apply unknown utility class `text-gray-text`
```

**원인**:

- Tailwind CSS 4에서는 커스텀 컬러를 `@theme` 블록 내에 정의해야 함
- `text-gray-text`, `text-gray-light`, `text-gray-lighter` 등이 정의되지 않음
- CSS 변수 `--gray-text`는 `:root`에 있지만, Tailwind이 인식하는 `--color-gray-text`가 없음

**해결**:

1. `@theme inline` 블록에 누락된 컬러 정의 추가:

   ```css
   @theme inline {
     /* 커스텀 컬러 - butterfly */
     --color-butterfly-50: #fef9e7;
     --color-butterfly-100: #fef3c7;
     --color-butterfly-700: #b7791f;
     --color-butterfly-800: #92400e;

     /* 커스텀 컬러 - gray */
     --color-gray-text: var(--gray-text);
     --color-gray-light: var(--gray-light);
     --color-gray-lighter: var(--gray-lighter);
   }
   ```

2. 전체 프로젝트에서 커스텀 컬러를 표준 Tailwind 컬러로 변경:
   - `text-gray-text` → `text-gray-800`
   - `text-gray-light` → `text-gray-600`
   - `text-gray-lighter` → `text-gray-500`
   - `bg-gray-text` → `bg-gray-800`
   - `border-gray-text` → `border-gray-800`
   - `ring-gray-text` → `ring-gray-800`

3. `.prose` 클래스는 CSS 변수 직접 사용:

   ```css
   .prose {
     color: var(--gray-text); /* @apply text-gray-text 대신 */
   }
   ```

4. 캐시 클리어 및 재시작:
   ```bash
   rm -rf .next
   npm run dev
   ```

**교훈**:

- Tailwind CSS 4는 이전 버전과 커스텀 컬러 정의 방식이 다름
- `@theme` 블록에 `--color-*` 형식으로 정의해야 Tailwind 유틸리티 클래스로 사용 가능
- CSS 변수만 정의하면 `var(--변수명)` 형태로만 사용 가능
- 표준 Tailwind 컬러 사용이 더 안전 (gray-800, gray-600 등)

2. 데이터 흐름 재설계:

### 이슈 3: 30번 문제 풀이 내용 오류

**증상**:

- 사용자: "풀이3이 없고 답이 다르다"
- 기존 풀이의 답: 243
- 올바른 답: 11

**원인**:

- 이전 세션에서 풀이를 작성할 때 사용자가 제공한 상세 풀이를 사용하지 않음
- 문제의 수학적 내용을 잘못 이해하고 임의로 작성
- step 파일 구조가 1, 2, 3, 4만 있고 3-1, 3-2, 3-3, 3-4 부분이 누락됨

**해결**:

1. 기존 step MDX 파일 전체 삭제:

   ```bash
   rm -rf content/problems/math-calculus/2025/regular/30/steps/*.mdx
   ```

2. 사용자가 제공한 정확한 풀이를 기반으로 8개 step 재작성:
   - `1.mdx`: 역함수 F(x)를 '부호까지' 확정
   - `2.mdx`: 교점 분석에 필요한 만큼만 f를 뒤집기
   - `3.mdx`: 교점 개수 g(m) 분석 (개요)
   - `3-1.mdx`: 오른쪽 구간 x > 2 분석
   - `3-2.mdx`: 왼쪽 구간 x < -2 분석 + 접선 조건
   - `3-3.mdx`: 중앙 구간 -2 ≤ x ≤ 2 분석
   - `3-4.mdx`: 전체 g(m) 합치기
   - `4.mdx`: 요구값 계산 (최종 답: 11)

3. 최종 답 계산 검증:
   ```
   g(a) × lim(m→b⁻) g(m) + g(b) × (ln b / b)²
   = 1 × 3 + 2 × 4
   = 3 + 8
   = 11 ✓
   ```

**교훈**:

- 사용자가 제공한 원본 자료를 항상 우선시해야 함
- 수학 문제는 정확성이 생명 - 임의로 추측하지 말 것
- 큰 작업 후에는 사용자와 검증 과정 필요
- 문제 풀이는 답부터 확인하고 역산해서 검증

### 이슈 4: git push 실패 (SSH 인증)

- remote 작업(push, pull)은 인증이 필요하므로 사용자가 직접 수행

## 논의 사항

### 1. URL 공유 링크 문제

**논의 내용**:

- 현재 구조에서는 문제 선택 시 URL이 변경되지 않음
- 특정 문제를 공유하거나 북마크할 수 없음

**해결 방안**:

- Option 1: Query string 사용 (`/?problem=30`)
  - 장점: 간단, 페이지 이동 없음
  - 단점: SEO에 불리, URL이 깔끔하지 않음

- Option 2: Next.js `shallow routing` + query params
  - 장점: 페이지 리로드 없이 URL 변경
  - 단점: App Router에서 아직 실험적 기능

- Option 3: 페이지 이동 방식으로 되돌림 (`/problems/30`)
  - 장점: RESTful, SEO 최적화, 공유 용이
  - 단점: 페이지 전환 발생, 현재 작업한 대화형 UI 포기

**결정 보류**: MVP 단계에서는 현재 구조 유지, 추후 query string 추가 고려

### 2. MDX 컴파일 시점

**논의 내용**:

- 현재: 클라이언트에서 사용자가 단서/풀이를 열 때 컴파일
- 대안: 서버에서 모든 MDX를 미리 컴파일

**선택한 방식**: 클라이언트 사이드 컴파일

**이유**:

- 초기 로딩 속도 최적화 (서버가 30개 문제의 모든 단서+풀이를 컴파일하면 느림)
- 사용자가 모든 문제를 보지 않을 수 있음 (낭비 방지)
- 컴파일 시간이 짧음 (1개 step당 100-200ms)

**트레이드오프**:

- 번들 크기 증가는 감수
- 로딩 UI로 사용자 경험 보완

### 3. 불연속 함수 그래프 시각화

**논의 내용**:

- 30번 문제는 F(x)의 불연속성이 핵심
- 텍스트 설명만으로는 이해하기 어려울 수 있음

**향후 고려사항**:

- Chart.js 또는 D3.js로 그래프 렌더링
- 또는 Desmos API 임베딩
- 또는 정적 이미지 추가

**현재**: 수식과 텍스트 설명으로 충분 (MVP 단계)

### 4. 풀이 단계 번호 체계

**논의 내용**:

- 현재: 1, 2, 3, 3-1, 3-2, 3-3, 3-4, 4
- 대안: 1, 2, 3, 4, 5, 6, 7, 8 (일련번호)

**선택한 방식**: 계층적 번호 (3-1, 3-2 등)

**이유**:

- 3단계가 복잡해서 하위 단계로 나눈 것이 명확히 드러남
- "구간별 분석"이라는 논리적 그룹핑
- 추후 다른 문제도 비슷한 패턴 사용 가능
