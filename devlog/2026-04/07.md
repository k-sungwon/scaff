## 기술적 결정 사항

### 1. Hybrid Architecture 채택

**결정**: Feature-Sliced Design + Micro Architecture 조합

**이유**:
- 도메인별 독립성 (entities/problem, entities/clue, entities/step)
- 비즈니스 로직 격리 (features/clue-reveal)
- 확장성 (새 과목/문제 추가 시 기존 코드 영향 최소화)
- 테스트 용이성 (각 엔티티를 독립적으로 테스트 가능)

**트레이드오프**:
- 장점: 코드 재사용성 높음, 의존성 방향 명확 (shared ← entities ← features)
- 단점: 파일 개수 증가 (30+ 파일), 초기 학습 곡선

**폴더 구조**:
```
shared/          # 공통 유틸리티, UI, 타입
entities/        # 도메인 모델 (problem, clue, step)
features/        # 비즈니스 로직 (clue-reveal)
app/             # Next.js 라우트 + API
```

### 2. Repository Pattern 도입

**결정**: 데이터 접근 로직을 인터페이스로 추상화

**이유**:
- 현재: 파일 시스템 (content/problems/)
- 미래: Supabase Storage
- 교체 시 UI 코드 수정 불필요

**트레이드오프**:
- 장점: 데이터 소스 교체 용이, 단위 테스트 시 Mock 사용 가능
- 단점: 레이어 하나 추가, 코드 양 증가

**코드 예시**:
```typescript
// Interface
export interface IProblemRepository {
  getProblemMeta(id: string): Promise<ProblemMeta | null>;
}

// FileSystem 구현체
export class FileSystemProblemRepository implements IProblemRepository {
  async getProblemMeta(id: string) {
    const path = this.resolveProblemPath(id);
    const result = await readFileIfExists(joinPath(path, "meta.json"));
    return safeParseProblemMeta(JSON.parse(result.content));
  }
}

// 향후 Supabase 구현체 추가 가능
export class SupabaseProblemRepository implements IProblemRepository {
  async getProblemMeta(id: string) {
    return await supabase.from("problems").select().eq("id", id).single();
  }
}
```

### 3. Server-side MDX 컴파일

**결정**: MDX를 서버에서 컴파일 후 클라이언트로 전송

**이유**:
- 클라이언트 번들 크기 500KB~1MB 감소
- KaTeX, remark, rehype 패키지를 서버에서만 사용
- 초기 로딩 속도 개선

**트레이드오프**:
- 장점: 빠른 초기 렌더링, 모바일 친화적
- 단점: API 응답 시간 증가 (첫 요청 시 컴파일 시간 ~100ms)

**성능 측정**:
```
GET /api/problems/.../clues
  compile: 1108ms (첫 요청, React.cache로 이후 0ms)
  render: 102ms
```

### 4. TanStack Query 도입

**결정**: 클라이언트 데이터 페칭에 TanStack Query 사용

**이유**:
- 자동 캐싱 (10분 staleTime)
- 로딩/에러 상태 자동 관리
- 자동 재시도 (2회)
- React Suspense 통합 가능

**트레이드오프**:
- 장점: 보일러플레이트 감소, 성능 개선
- 단점: 라이브러리 의존성 추가 (~50KB gzipped)

**코드 예시**:
```typescript
// features/clue-reveal/hooks/use-clues.ts
export function useClues(problemId: string) {
  return useQuery({
    queryKey: ["clues", problemId],
    queryFn: () => getClues(problemId),
    staleTime: 1000 * 60 * 10, // 10분 캐시
    retry: 2,
  });
}

// 사용처
const { data: clues, isLoading, error } = useClues(problemId);
```

### 5. Zod 스키마 검증

**결정**: JSON 파일 읽기 후 Zod로 런타임 검증

**이유**:
- 잘못된 meta.json 조기 발견
- TypeScript 타입과 검증 규칙 동기화
- 안정적인 운영 환경

**트레이드오프**:
- 장점: 런타임 안정성 보장, 타입 추론 자동
- 단점: 검증 오버헤드 (~1ms per file)

**코드 예시**:
```typescript

  year: z.number().min(1900),

    return ProblemMetaSchema.parse(data);
```

### 6. Result Pattern

**결정**: 파일 읽기 결과를 명시적 Result 타입으로 반환

**이유**:
- 성공/실패/파일없음을 명확히 구분
- try-catch 중첩 제거
- 타입 안전한 에러 처리

**코드 예시**:
```typescript
// shared/lib/fs/async-fs.ts
export type FileReadResult =
  | { status: "success"; content: string }
  | { status: "not_found"; path: string }
  | { status: "error"; error: Error };

export async function readFileIfExists(filePath: string): Promise<FileReadResult> {
  try {
    const content = await fs.readFile(filePath, "utf-8");
    return { status: "success", content };
  } catch (error: any) {
    if (error.code === "ENOENT") {
      return { status: "not_found", path: filePath };
    }
    return { status: "error", error };
  }
}
```

### 7. Parallel File Loading

**결정**: Promise.all로 여러 파일 동시 읽기

**이유**:
- 5개 단서 파일 순차 읽기 → 병렬 읽기로 변경
- 예상 성능 개선: 5배 (500ms → 100ms)

**코드 예시**:
```typescript
// shared/lib/fs/async-fs.ts
export async function readFilesParallel(paths: string[]): Promise<FileReadResult[]> {
  return Promise.all(paths.map(readFileIfExists));
}

// 사용처
const cluePaths = clues.map(c => joinPath(basePath, c.content));
const results = await readFilesParallel(cluePaths);
```

## 발생한 이슈

### 이슈 1: Next.js 16 params 타입 오류

**증상**:
```
Type '{ params: { id: string; }; }' is not assignable to
type '{ params: Promise<{ id: string; }>; }'
```

**원인**: Next.js 16부터 Route Handler의 params가 Promise로 변경됨

**해결**:
```typescript
// Before
export async function GET(
  request: NextRequest,
  { params }: { params: { id: string } }
) {
  const { id } = params;
}

// After
export async function GET(
  request: NextRequest,
  { params }: { params: Promise<{ id: string }> }
) {
  const { id } = await params;  // await 추가
}
```

**교훈**: Next.js 메이저 버전 업그레이드 시 breaking changes 문서 필수 확인

### 이슈 2: Zod 타입 정의 충돌

**증상**:
```
Property 'errors' does not exist on type 'ZodError<unknown>'
```

**원인**: Zod 버전에 따라 ZodError API가 다름

**해결**: 에러 객체 전체를 로깅하도록 수정
```typescript
// Before
catch (error) {
  if (error instanceof z.ZodError) {
    console.error("Validation failed:", error.errors);
  }
}

// After
catch (error) {
  console.error("Parsing error:", error);
}
```

**교훈**: 외부 라이브러리 타입 사용 시 버전 확인 및 타입 가드 신중히 작성

### 이슈 3: Namespace vs Type Export 충돌

**증상**:
```
Cannot redeclare exported variable 'Clue'
```

**원인**: entities/clue/index.ts에서 타입과 네임스페이스 이름 중복

**해결**:
```typescript
// Before
export type { Clue } from "./model/clue.types";
export const Clue = { API: ClueAPI };  // 충돌!

// After
export type { Clue } from "./model/clue.types";
export const ClueNamespace = { API: ClueAPI };
```

**교훈**: Barrel export 시 타입과 값의 네이밍 충돌 주의

### 이슈 4: ProblemMeta 타입 불일치

**증상**:
```
Type 'string[]' is not assignable to type 'Concept[]'
Property 'clues' is missing in type '...'
```

**원인**: lib/constants.ts의 MVP_PROBLEMS가 구 타입 정의로 작성됨

**해결**: 모든 필수 필드 추가
```typescript
// Before
{
  id: "2025-math-regular-21",
  concepts: ["미적분", "극한"],  // Wrong
}

// After
{
  id: "2025-math-regular-21",
  concepts: [],                  // Correct
  tags: ["미적분", "극한"],
  clues: [],
  steps: [],
  commonMistakes: [],
  estimatedTime: 15,
}
```

**교훈**:
- 타입 정의 변경 시 기존 데이터도 함께 업데이트
- Zod 스키마를 먼저 정의하고 타입을 추론하는 방식이 더 안전
